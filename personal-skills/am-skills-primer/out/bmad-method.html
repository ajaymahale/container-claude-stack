BMAD Method — vs GSD

Breakthrough Method of Agile AI-Driven Development ("Build More Architect Dreams"), by bmad-code-org. MIT, free. V6 = skills-architecture. Not installed locally — researched from source.

Source: github.com/bmad-code-org/bmad-method · docs.bmad-method.org · install: npx bmad-method install

One-line: BMAD simulates an entire agile product team as 6 named AI personas (PM, Architect, Dev, UX, Analyst, Tech-Writer) who collaborate through structured workflows from brainstorm → PRD → architecture → stories → code. Traditional AI "does the thinking for you"; BMAD personas facilitate your thinking. The framework is scale-adaptive (bug-fix → enterprise) and artifact-driven (PRD, Architecture, Epics, Stories — sharded for context).

Two trigger types: workflow (just type the code, e.g. PRD) vs conversational (code + your description, e.g. WD write a deploy guide).

Workflows by phase

meta   bmad-help
Context-aware "what's next" router. Ask bmad-help I finished architecture, what now?. The on-ramp — GSD's /gsd-progress equivalent.
analysis   Analyst + Tech Writer — BP DR TR CB DP
Brainstorm / Product Brief / PRFAQ (press-release FAQ), Market/Domain/Technical Research, Document Project (full-scan or deep-dive of an existing codebase). Output: briefs + research that feed the PRD. Skills: bmad-brainstorm·bmad-product-brief·bmad-prfaq·bmad-document-project.
plan   PM + UX — PRD CE CU
Create / Edit / Validate PRD (with headless validation + report), Create Epics & Stories, Create UX Design (design directions, wireframes, key screens, validation). Skills: bmad-create-prd·bmad-edit-prd·bmad-validate-prd·bmad-create-epics-and-stories·bmad-ux.
solutioning   Architect — CA IR
Create Architecture with adversarial review (stress-test the design) + Implementation Readiness gate (are we ready to build?) + Generate Project Context. This is BMAD's de-risk-upfront phase — "architect dreams". Skills: bmad-create-architecture·bmad-check-implementation-readiness·bmad-generate-project-context.
implementation   Developer — DS QD CR SP IN
Dev Story (build one story), Quick Dev (scale-adaptive small task), Dev Auto (autonomous dev loop), Code Review, Sprint Planning/Status, Create Story, QA generate E2E tests, Retrospective, Investigate / Forensic (post-mortem), Correct Course, Checkpoint Preview.

Distinctive mechanisms

MechanismWhat it is
Party ModeMultiple personas co-present in one session to collaborate / debate. GSD has no equivalent — its agents are spawned-per-job, not co-present.
Scale-Domain-AdaptiveAuto-adjusts planning depth: quick-dev for a bug fix, full solutioning for enterprise. Mirrors GSD /gsd-quick vs /gsd-execute-phase.
Adversarial ReviewArchitecture gets stress-tested before it stands. GSD's analog is plan-time cross-AI review.
Document ShardingLarge artifacts (PRD, Architecture) split into shards for context management. GSD uses per-phase files instead.
Implementation-Readiness GateExplicit "are we ready to build?" checkpoint before dev. GSD's analog = plan-checker + decision-coverage gate.
Web BundlesOffload upfront planning (Brainstorm/PRD/UX/Research) to Gemini Gems / ChatGPT GPTs — flat-rate subscription vs metered IDE tokens. Bring artifacts back to IDE. No GSD equiv.
ModulesCore BMM + BMad Builder (custom agents), Test Architect Enterprise (TEA), Game Dev (BMGD), Creative Intelligence (CIS).

For your stack (.NET backend, legacy reverse-map, already on GSD)

Low direct fit. BMAD is greenfield / web-product shaped (PRFAQ, UX, market research, product brief). Your wow-pic work is incremental engineering on an established system, deep in GSD's durable phase/milestone/verification loop — that's the better tool for it. Don't migrate.
Transferable ideas worth stealing:
• Party Mode — persona debate in one session for hard design calls (your DSL binding decisions)
• Adversarial architecture review — pair with Addy's doubt-driven-development for the LegacyMapperGenerator
• Document sharding — if any PRD/SPEC balloons past context
• Web bundles — offload big upfront planning to a flat-rate Gemini/ChatGPT sub instead of metered IDE tokens
• Implementation-Readiness gate — a lighter analog to GSD's plan-checker for greenfield spikes

If you ever go greenfield (new service from scratch), BMAD is the stronger starting frame; GSD is the stronger delivery frame. They're complementary by lifecycle stage, not substitutes.
