Superpowers — vs GSD

A complete development methodology for coding agents, built on composable skills that auto-trigger. By Jesse Vincent (obra / Prime Radiant). ● ACTIVE in this session (v6.0.0, on disk).

Source: github.com/obra/superpowers · disk: ~/.claude/plugins/cache/claude-plugins-official/superpowers/6.0.0/skills/ (14 skills)

One-line: Don't jump to code. Socratic-brainstorm a spec → show it in digestible chunks → sign off → write a plan "clear enough for an enthusiastic junior engineer with poor taste, no judgement, no project context, and an aversion to testing" → say "go" → subagent-driven development: fresh agents work each task, inspected + reviewed, running autonomously for hours. True red/green TDD, YAGNI, DRY. Skills fire automatically — "your agent just has Superpowers."

The 14 skills — by phase

auto = skill auto-fires on trigger (all of Superpowers is auto/model-invoked). meta = governs how skills are used.

meta   using-superpowers
The spine. Forces a skill check before ANY response, including clarifying questions. "If there's even a 1% chance a skill applies, you MUST invoke it." This is what makes Superpowers ambient instead of slash-driven. It's running right now.
define   brainstorming
MANDATORY before any creative work — features, components, functionality, behavior changes. Explores user intent + requirements + design before implementation. The Socratic spec-tease.
plan   writing-plans
Have a spec/requirements for a multi-step task, before touching code. Plan written junior-engineer-clear, emphasizing red/green TDD + YAGNI + DRY.
plan   executing-plans
Execute a written plan in a separate session with review checkpoints.
isolate   using-git-worktrees
Ensure an isolated workspace via a worktree before feature work or plan execution. (Worktrees are broken in your env — .planning gitignored — same reason you run GSD sequential-on-main.)
isolate   dispatching-parallel-agents
2+ independent tasks with no shared state / sequential deps. Fan out.
execute   subagent-driven-development
Signature move. Execute a plan's independent tasks in the current session by spawning fresh subagents per task, inspecting + reviewing each, continuing forward. Can run autonomously for hours without deviating from the plan.
execute   test-driven-development
Strict. Red-Green-Refactor. Will delete prematurely-written code to hold the line. Use before any implementation code.
debug   systematic-debugging
Any bug / test failure / unexpected behavior — before proposing fixes. Scientific method.
review   requesting-code-review
Completing tasks / major features / before merging — verify work meets requirements.
review   receiving-code-review
Receiving feedback before implementing it, esp. when unclear/technically questionable — requires technical evaluation, not blind application.
verify   verification-before-completion
Before claiming done/fixed/passing — must run verification commands + confirm output, never "seems right." Commits/PRs gated on it.
ship   finishing-a-development-branch
Implementation complete, tests pass — decide how to integrate (merge strategy, etc.).
meta   writing-skills directly useful
Skills-that-write-skills. Creating/editing/verifying skills before deployment. You author am-* skills — this is the craft reference. No GSD equivalent.

For your stack (.NET backend, legacy reverse-map, already on GSD)

It's GSD's closest sibling — overlap is high. Both: subagents + worktrees + plans + TDD + verification. Don't run both methodologies' planning in parallel — pick one spine. You're on GSD (durable memory, milestone governance) → stay on GSD. Superpowers running ambiently here mostly reinforces GSD's discipline, it doesn't replace it.
Live tension to name: Superpowers mandates brainstorming-before-creative-work + strict TDD (deletes premature code). Ponytail (your active mode) says YAGNI / minimal / shortest-path. They coexist but pull opposite directions on ceremony — when they conflict, user instructions win (Superpowers' own priority rule), so Ponytail / explicit asks override the mandate.
Transferable value over GSD:
• verification-before-completion rigor — "run the command, show the output" before claiming done. Pairs with your post-phase test gate + byte-identical snapshot gates.
• systematic-debugging — scientific-method debug loop (complements /gsd-debug).
• writing-skills — directly useful; you author am-* skills, this is the craft reference GSD lacks.
• Two-stage review pattern (spec-compliance then code-quality) — graftable onto /gsd-code-review.

Where GSD stays ahead: memory/state (MemPalace, threads, capture/resume across sessions), milestone lifecycle, goal-backward hard-blocker verification. Superpowers is stateless by design.
