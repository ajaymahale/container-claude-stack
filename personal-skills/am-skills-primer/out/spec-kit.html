🌱 GitHub Spec Kit — vs GSD

Spec-Driven Development toolkit (the "Specify" CLI). Spec becomes executable — define what+why, refine through structured phases, agent implements. Anti-"vibe coding." Agent-agnostic (30+ tools). Not installed locally — researched from source.

Source: github/spec-kit · github.github.io/spec-kit · GitHub Blog

Install + init (needs uv): 
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@vX.Y.Z
specify init my-project --integration claude-code   # or copilot, gemini-cli, codex ...
cd my-project                                        # then launch your agent here
Commands surface as /speckit.* (Claude/Copilot/Gemini), $speckit-* (Codex skills), /agents (Copilot CLI). specify integration list → all 30+. 
  constitution → [clarify] → specify → plan → tasks → [analyze] → implement → converge ↺
    governing           what/why    how      units     coverage     build      loop-to-spec

The pipeline — 7 core commands

#CommandDoes
0constitution /speckit.constitutionFirst. Project governing principles (quality, testing, UX, perf) guiding ALL later work.
·clarify* /speckit.clarifyClarify underspecified areas. Run before plan. (was /quizme)
1specify /speckit.specifyDefine what+why — requirements + user stories. Not the tech stack.
2plan /speckit.planTechnical implementation plan w/ chosen stack + architecture.
3tasks /speckit.tasksActionable task list from the plan.
·analyze* /speckit.analyzeCross-artifact consistency + coverage. After tasks, before implement.
4implement /speckit.implementExecute all tasks, build the feature per the plan.
5converge /speckit.convergeAssess codebase vs spec/plan/tasks → append remaining work as new tasks. The loop.

/speckit.taskstoissues (tasks → GitHub issues) · /speckit.checklist (quality checklists — "unit tests for English"). Artifacts: .specify/memory/constitution.md, specs/ (spec/plan/tasks). * = optional.

Customization — the priority stack (Spec Kit's superpower)

PriLayerPurpose
1.specify/templates/overrides/one-off project-local tweaks
2Presetscustomize how it works — override templates/commands (compliance, DDD, language, test-first)
3Extensionsadd new capabilities — new commands/templates (Jira, post-impl review, V-Model)
4Core defaultsbuilt-in SDD commands + templates

Resolved top-down at runtime, first match wins. Bundles = role-based package (PM, BA, security, dev) of extensions+presets+workflows in one versioned bundle.yml — specify bundle search/info/install. Idempotent, project-confined, offline.

Development modes

0-to-1 greenfield — generate from scratch
Start with high-level requirements → generate specs → plan steps → build production-ready app.
exploration creative — parallel implementations
Explore diverse solutions, multiple tech stacks & architectures, experiment with UX patterns.
brownfield iterative enhancement your territory
Add features iteratively, modernize legacy systems, adapt processes. spec-persistence + evolving-specs loop keeps tooling updates separate from artifact evolution.

For your stack (.NET backend, legacy reverse-map, already on GSD)

Low migration pressure. Your established codebase is deep in GSD's spec→plan→execute→verify loop with durable memory + milestone tracking + goal-backward rigor — thicker than Spec Kit for project work. Stay GSD as spine.
Transferable ideas worth stealing:
• constitution as a governed artifact — you already keep principles in CLAUDE.local.md (byte-identical gates, source-driven, no-vocab-leakage, Aspire HTTP profile). Promote to a checked-in constitution.md every spec references.
• converge loop — mirrors your byte-identical generator snapshot gate; name it ("keep going till snapshot green").
• brownfield / iterative-enhancement — your exact scenario; if you ever trial Spec Kit, this is the mode, not 0-to-1.
• extensions/presets — want DDD-flavored or compliance-gated SDD? presets are the lever.

Spec Kit genuinely wins when you need the same spec on Copilot and Claude (portable unit), or want the extension/bundle ecosystem around spec→code.
